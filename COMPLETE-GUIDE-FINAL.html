1. Core Data Structures

1.1 - Student Structure

struct Student
Stores individual student information
struct Student {
    int sl;         // Serial number (1-45)
    string id;      // Student ID (e.g., "C253027")
    string name;    // Full name

    // Constructor
    Student(int serial, const string &studentId, const string &studentName)
        : sl(serial), id(studentId), name(studentName) {}
};
Why These Fields?
‚Ä¢ sl - Maintains roster order for display (1-45)
‚Ä¢ id - Unique identifier for attendance tracking
‚Ä¢ name - Display in UI and CSV files

Size Estimate: ~80 bytes per Student
‚Ä¢ int sl = 4 bytes
‚Ä¢ string id = ~24 bytes (object) + 8 bytes (data)
‚Ä¢ string name = ~24 bytes (object) + 20-40 bytes (data)
Storage Container: vector<Student> students;

Why vector instead of map?
‚úÖ Preserve roster order (SL 1-45 sequence)
‚úÖ Sequential iteration common (display all students)
‚úÖ No frequent lookups by ID needed
‚úÖ Linear search acceptable for 45 students (O(45) ‚âà constant)
‚ùå Map would be overkill for this size

1.2 - Course Structure

struct Course
Stores course metadata and semester schedule
struct Course {
    string code;           // Course code (e.g., "EEE-1121")
    string title;          // Full title
    int weeklyClasses;     // Classes per week
    int totalSemesterClasses;  // Total classes in 16-week semester

    // Constructor

           int weekly, int total)

Why map instead of vector?
‚úÖ Fast O(log n) lookup by course code
‚úÖ Automatic alphabetical sorting
‚úÖ Key-value association clearer (code ‚Üí Course)
‚úÖ No duplicates needed
‚ùå Vector would require O(n) linear search
Special Case - GEEM-1101:

All other courses: totalSemesterClasses = 48
GEEM-1101: totalSemesterClasses = 32

Why? 16 weeks √ó 2 classes/week = 32 (shorter course)
Other courses: 16 weeks √ó 3 classes/week = 48

1.4 - AttendanceRecord Structure

struct AttendanceRecord
Records a single attendance event for one student
struct AttendanceRecord {
    string studentId;  // Which student (e.g., "C253027")
    string courseCode; // Which course (e.g., "EEE-1121")
    string date;       // Date (DD-MM-YYYY format)
    string day;        // Day of week
    string timeSlot;   // Start time
    bool present;      // true = Present, false = Absent

    // Constructor
    AttendanceRecord(const string &id, const string &course,
                     const string &dateStr, const string &dayStr,
                     const string &time, bool isPresent)
        : studentId(id), courseCode(course), date(dateStr),
          day(dayStr), timeSlot(time), present(isPresent) {}
};
Storage Container: vector<AttendanceRecord> attendanceRecords;

Why vector?
‚úÖ Append-only pattern (new attendance added to end)
‚úÖ Filtering by student/course using STL algorithms
‚úÖ Size varies (can grow to 1000s of records)
‚úÖ CSV export iterates sequentially
Size Estimate: ~150 bytes per AttendanceRecord
‚Ä¢ 5 strings @ ~30 bytes each = 150 bytes
‚Ä¢ 1 bool = 1 byte
‚Ä¢ Total: ~151 bytes

1000 records = ~150 KB of memory

Why floating-point for percentages?
(attended / total) * 100 with integers = 0 for small values!
Example: (2 / 48) * 100 = 0 * 100 = 0 (integer division)
(2.0 / 48.0) * 100.0 = 0.0417 * 100.0 = 4.17 (correct)

static_cast<double>():
Explicitly converts integer to double before division
static_cast<double>(attended) / total * 100.0

4. Navigation Patterns

Standard Navigation Convention
Input Options Across All Menus:
  • [0] = Back to previous menu
  • [9] = Jump directly to main menu
  • [R] = Reset/Clear current operation
  • 1-N = Menu option numbers
Return Value Pattern: 
bool selectPeriodAndMark(...) {
    if (periodInput == "0") return false;  // [0] = Back (stay in loop)
    if (periodInput == "9") return true;   // [9] = Main menu (exit completely)
    // ... rest of function
    return false;  // Default: stay in menu
}

// In calling function:
bool exitToMain = selectPeriodAndMark(...);
if (exitToMain) return;  // Propagate [9] to exit all nested menus
Why boolean?
‚Ä¢ true = Exit to main menu (bubble up through all levels)
‚Ä¢ false = Return to previous menu (one level up)
‚Ä¢ Allows [9] to exit nested menus in single press

Lambda Usage in Reset Operations
// RESET FUNCTIONALITY - Remove specific records

    remove_if(attendanceRecords.begin(), attendanceRecords.end(),
             [&](const AttendanceRecord &r) {
                 for (const auto &period : selectedPeriods) {
                     if (r.courseCode == period.courseCode &&

Lambda Breakdown:

1. Capture List [&]:

Allows lambda to access selectedPeriods and dateInput

2. Parameter (const AttendanceRecord &r):
Each record passed to lambda for testing

3. Return Value (bool):
true = Remove this record
false = Keep this record

Why lambda instead of separate function?
‚Ä¢ Inline logic (no function declaration needed)

7.3 - Range-Based For Loops

const auto & Pattern
// PATTERN:

    // Process student
}

// BREAKDOWN:
// 'for' - Range-based loop
// 'const' - Cannot modify student (read-only)
// 'auto' - Type deduced automatically (Student)
// '&' - Reference (no copy, efficient)
// Alternative: 'auto' alone copies each element (slower!)
Performance Comparison:

‚úÖ No copying (reference)

‚úÖ Type-safe
