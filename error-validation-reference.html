4. Buffer Management Scenarios (Complete Guide)

The Golden Rule: cin >> leaves '\n' in buffer, getline() consumes it. Must call cin.ignore() before getline() if previous input was cin >>.
Scenario Input Sequence Buffer State Required Action Code Location
A 1. cin >> periodChoice;
2. getline(cin, dateInput); 		'\n' left by cin >> MUST cin.ignore(); between steps Single period mode
B 1. getline(cin, multiInput);
2. getline(cin, dateInput); 		Empty (getline consumed '\n') NO ignore needed Multi-period date input
C 1. cin >> periodInput; (reads "7")
2. cin.ignore(); (clears '\n')
3. getline(cin, multiInput); 		'\n' after "7", cleared by ignore ignore() AFTER cin >> before getline Entering multi-period mode
D
(BUG!)		 1. cin >> "7" + cin.ignore()
2. getline(multi) (buffer clean)
3. cin.ignore() ‚Üê WRONG!
4. getline(date) ‚Üê Skips user input! 		Empty after step 2, no '\n' to ignore SKIP second ignore (use flag) WHY isMultiPeriodMode exists

isMultiPeriodMode Flag Implementation

Code Section isMultiPeriodMode Value Action Taken
Initial declaration:
bool isMultiPeriodMode = false;		 false (default) Assume single period mode
If user enters "7":
if (periodInput == "7") {
  isMultiPeriodMode = true;
}		 true Mark that we entered multi-period
Before date input:
if (!isMultiPeriodMode) {
  cin.ignore();
}		 If false ‚Üí ignore
If true ‚Üí skip ignore		 Only ignore if we didn't use getline
Date getline:
getline(cin, dateInput);		 N/A Works correctly either way
üî• Why This Matters:
Without the isMultiPeriodMode flag, multi-period mode would double-ignore the buffer:
1. cin.ignore() after entering "7" (clears '\n' from cin >>)
2. getline() for multi-period input (buffer now clean)
3. cin.ignore() before date input ‚Üê PROBLEM! Consumes user's actual date input
4. getline() for date ‚Üê Gets empty string or next line

Result: User types date, program skips it. Critical bug prevented by flag!
